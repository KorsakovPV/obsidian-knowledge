# Правила доступности действий: Сделка и Оффер (КП)

#research #project

> Правила реализованы паттерном resolver'ов в `app/services/deal_actions/` и
> `app/services/offer_actions/`. Общий контекст — в [[Architecture]] (раздел
> «Согласование (approvals) и действия»), сущности — в [[Domain Model]].
> Связанная модель прав — [[LKM Role Model]].

## Состояния согласования

| Состояние | Условие |
|---|---|
| `MISSING` | Согласование отсутствует |
| `DRAFT` | Согласование создано, не отправлено |
| `REQUIRED` | Согласование отправлено, ожидается ответ (`status=pending`) |
| `APPROVED` | Получен ответ — одобрено (`status=answered`, `decision=approve`) |
| `REJECTED` | Получен ответ — отклонено (`status=answered`, `decision=reject`) |
| `CANCELED` | Согласование отменено (`status=canceled`) |
| `UNKNOWN` | Не соответствует ни одному из перечисленных вариантов |

## Оффер (КП)

### Связь действий с permissions

В текущей матрице прав нет отдельных permissions для редактирования и удаления КП.
Поэтому backend-guard для действий `edit` и `delete` оффера использует permission
`create_kp`: пользователь, которому разрешено создавать КП, может выполнять эти
действия, если они не заблокированы состоянием согласования.

### Редактирование оффера

| Состояние согласования | Доступно | Причина |
|---|:---:|---|
| `MISSING` | ✅ | |
| `DRAFT` | ✅ | |
| `REQUIRED` | ❌ | Согласование на рассмотрении — нельзя менять параметры пока ожидается ответ |
| `APPROVED` | ✅ ⚠️ | Разрешено, но любое изменение автоматически сбрасывает согласование в `DRAFT` |
| `REJECTED` | ✅ | |
| `CANCELED` | ✅ | |
| `UNKNOWN` | ❌ | Неизвестное состояние — блокируем во избежание непредсказуемого поведения |

**Сброс согласования при редактировании `APPROVED` оффера**

При сохранении любого изменения оффера в состоянии `APPROVED` согласование сбрасывается:
- `status` → `DRAFT`
- `decision` → `null`
- `decision_comment` → `null`
- `decided_at` → `null`

После сброса оффер требует повторной отправки на согласование перед тем, как его можно будет снова использовать в имплементации.
